## 联调机制概述

为了将整个系统从“静态 Mock”状态提升到由 AI 驱动决策的“多智能体协作”状态，我们实装了真实的 Server-Sent Events (SSE) 流式应答适配器，并在中央交互舱（B 区）与智能体监控区（C 区）之间建立起了语义指令路由拦截体系。

> 系统不再依靠前端界面编写死锁的按钮或正则判断，而是由大模型在实时推理后自发决定是否通过 `[EXECUTE: command]` 标签下发执行动作，驱动底层子进程物理执行。

---

## 真实流式适配器设计 (SSE Stream Adapter)
- **参数读取**：系统实时从持久化配置库 `useSettingsStore` 中检索 `apiKey`、`apiBaseUrl` 与 `defaultModel`。
- **SSE 通信管道**：
  - 发送 HTTP POST 请求，配置 `stream: true` 开启大模型流式响应。
  - 通过 `response.body.getReader()` 捕获 `ReadableStream`。
  - 使用 `TextDecoder` 对字节流进行解码，逐行检索以 `data: ` 开头的 JSON 结构块。
- **打字机交互（Typewriter Visuals）**：
  - 渲染层在用户发送消息时，即刻向消息队列中塞入一条空内容的机器回应占位气泡。
  - 在大模型数据包流式吐出期间，利用 `onChunk` 增量回调实时更新该消息气泡的 `content` 状态，驱动气泡字符平滑流出。

## 语义指令路由解析 (Semantic Router)

语义指令路由系统是多智能体协作的核心大脑，其运行规程与数据链路由以下三个环节组成：

### 1. 动态上下文感知注入 (System Context Injection)
在向模型发起提问时，系统会将全局扫描得到的 `workspaceFiles`（项目相对路径、文件大小以及前 150 字符的内容摘要）自动拼装作为系统的背景知识（Context）注入到 System Prompt 中：
- 赋予 Harri 对当前本地项目的物理文件树结构、命名约定、技术栈的完整认知。
- 模型在获得上下文后，能够精准定位需要修改的文件。

### 2. 标签拦截正则表达式
在应答结束时，路由器会通过以下正则表达式对完整的回答文本进行捕获匹配：
```typescript
const executeRegex = /\[EXECUTE:\s*([^\]]+)\]/i;
```

> **指令标签规范**：若大模型推理后认为应当运行指令（如编译、分析或测试），需在回复尾部带上特定标识，例如 `[EXECUTE: npm run build]`。

### 3. 数据总线路由派发
- 正则匹配成功后，提取指令及参数。
- 派发全局事件：
  ```typescript
  eventBus.emit('TASK_TRIGGER', { type, description })
  ```
- 右侧 C 栏的 [SubagentMonitor](file:///d:/MiMo%20One/src/components/Subagent/SubagentMonitor.tsx) 组件接收该事件，并调用本地物理执行器拉起子进程，实时将带有 ANSI 控制码的 stdout 输出呈递给用户。

## 环境优雅降级策略 (Graceful Degradation)

在未配置 API Key 的无凭证状态下（如初次启动），适配器将自动静默退化到仿真降级模式：
- 模拟生成带有安全提示的打字机流式字符，防止系统发生断连异常。
- 提示用户可在对话中直接以中文发送“编译”或“分析”等词汇，绕过大模型推理直接以硬编码 TASK_TRIGGER 的形式唤起物理执行器。
